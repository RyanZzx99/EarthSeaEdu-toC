# 建档流程方案

## 1. 本文档目的

本文档用于沉淀“首页新增固定问卷建档链路”的当前方案，便于在后续会话中直接继续讨论或落地。

本期只讨论方案，不落代码。

---

## 2. 已确认的产品边界

### 2.1 首页入口

首页保留原有 AI 自由对话建档链路，不做删除。

首页新增一个并列按钮，进入新的固定问卷建档链路。

建议首页最终有两个入口：

- `AI 自由建档`
- `标准问卷建档`

### 2.2 新旧链路隔离原则

原 AI 对话链路完全保留。

新固定问卷链路：

- 不复用 `ai_chat_sessions`
- 不复用 `ai_chat_messages`
- 不复用 `ai_chat_profile_results`

即：

- 历史会话隔离
- 消息记录隔离
- 结果记录隔离

新链路只共享两类能力：

- 正式业务表结构
- 六维图评分/生成能力

### 2.3 每题立即写库

新链路中，用户每答一题都立即写数据库。

本期接受以下口径：

- 每答一题，除了写新链路自己的会话/消息/答案表
- 同时将累计答案投影为正式业务表快照，并立即写入正式业务表

即：

- 正式业务表会随着问卷作答过程逐步变成“当前已答部分”的状态
- 不再额外为了避免与旧链路互抢而引入中间快照隔离层

### 2.4 中途退出也生成六维图

当用户：

- 全部答完
- 答到一半关闭问卷
- 主动退出聊天框
- 页面关闭或离开页面

都要触发一次六维图生成。

### 2.5 多国家 / 多专业偏好本期就做

当前 `student_basic_info` 中的：

- `CTRY_CODE_VAL`
- `MAJ_CODE_VAL`

需要从“主表直接存一个值”升级为“拆分子表，多条偏好记录”。

但由于旧 AI 链路本期必须完全保留，且旧链路当前仍依赖这两个旧字段，所以本期方案为：

- 新链路的数据源改为新子表
- `student_basic_info.CTRY_CODE_VAL / MAJ_CODE_VAL` 暂不物理删除
- 这两个旧字段保留为兼容镜像字段
- 新链路写新子表后，同步第一志愿回旧字段

这条规则已确认。

## 3. 新链路总体形态

### 3.1 交互形态

新链路仍保留聊天框形式，但不是自由输入对话，而是固定问卷驱动。

表现形式：

- 助手气泡显示问题
- 下方显示固定选项 / 多选卡片 / 条件表单
- 用户提交后，聊天区插入一条“用户回答摘要”
- 再显示下一题

即：

- 形式是聊天
- 逻辑是问卷
- 提问不再由 AI 参与

### 3.2 后端逻辑形态

新链路是“配置驱动问卷链路”，不是“AI 驱动自由对话链路”。

推荐结构：

- 问卷配置：由后端定义问题、选项、分支、校验规则
- 会话状态：记录当前做到哪一题
- 答案存储：记录每题当前生效答案
- 消息存储：单独存聊天展示历史
- 结果存储：单独存本链路生成的六维图结果

## 4. 新链路数据表方案

为符合当前项目已有表结构风格，建议新表命名使用 `student_profile_*` 风格。

建议新增 4 张主表：

- `student_profile_guided_sessions`
- `student_profile_guided_messages`
- `student_profile_guided_answers`
- `student_profile_guided_results`

同时新增 2 张正式业务子表：

- `student_basic_info_target_country_entries`
- `student_basic_info_target_major_entries`

---

## 5. 表结构风格约束

新表字段风格遵循当前项目已有表风格：

- 主键：`id BIGINT`
- 用户归属字段：`student_id CHAR(36)`
- 会话业务键：`session_id CHAR(36)`
- 状态字段：`session_status / current_stage / result_status`
- 审计字段：
  - `remark`
  - `created_by`
  - `updated_by`
  - `create_time`
  - `update_time`
  - `delete_flag`

建议统一使用：

- `delete_flag CHAR(1)`，`1` 有效，`0` 删除
- `create_time / update_time`
- 需要承载结构化内容时才用 `JSON`

## 6. 新链路核心表说明

## 6.1 `student_profile_guided_sessions`

### 作用

这张表不是用来存消息，而是用来存“这一轮固定问卷建档会话本身”。

它负责：

- 标识一轮问卷会话
- 记录当前进行到哪一题
- 记录当前阶段
- 记录答案版本号
- 支撑中途退出恢复
- 支撑避免重复生成六维图

### 建议字段

- `id`
- `session_id`
- `student_id`
- `questionnaire_code`
- `questionnaire_version`
- `current_question_code`
- `current_question_index`
- `session_status`
- `current_stage`
- `version_no`
- `last_generated_version_no`
- `started_at`
- `finished_at`
- `exited_at`
- `remark`
- `created_by`
- `updated_by`
- `create_time`
- `update_time`
- `delete_flag`

### 字段说明

- `questionnaire_code`
  - 当前问卷编码，例如 `student_profile_guided_v1`
- `questionnaire_version`
  - 当前问卷版本
- `current_question_code`
  - 当前题编码，例如 `Q7`
- `current_question_index`
  - 当前题序号
- `session_status`
  - 建议值：`active / completed / exited / failed`
- `current_stage`
  - 建议值：`collecting / build_ready / scoring / generated / failed`
- `version_no`
  - 当前会话累计答案版本号
- `last_generated_version_no`
  - 最近一次已生成六维图时，对应的答案版本号

### 建议字段

- `session_id`
- `student_id`
- `sequence_no`
- `message_role`
- `message_kind`
- `question_code`
- `content`
- `payload_json`
- `remark`
- `created_by`
- `updated_by`
- `create_time`
- `update_time`
- `delete_flag`

### 字段说明

- `message_role`
  - 建议值：`assistant / user / system`
- `message_kind`
  - 建议值：`question / answer / notice / generation_status`
- `question_code`
  - 该消息关联的题目编码
- `content`
  - 直接用于聊天气泡展示的文本
- `payload_json`

### 设计原则

- 这张表负责“聊天展示历史”
- 不负责“答案真值”
- 当前答案真值由 `student_profile_guided_answers` 负责

## 6.3 `student_profile_guided_answers`

### 作用

这张表用于存“每道题当前生效答案”，不是完整审计历史。

即：

- 每道题在当前 session 中只有一条最新有效答案
- 用户修改同一题时，更新这条记录

如果未来要做严格审计，可再新增独立日志表；本期不做。

### 建议字段

- `id`
- `session_id`
- `student_id`
- `question_code`
- `question_type`
- `module_code`
- `answer_json`
- `answer_display_text`
- `version_no`
- `remark`
- `created_by`
- `updated_by`
- `create_time`
- `update_time`
- `delete_flag`

### 关键字段说明

#### `answer_json`

存“标准化答案对象”，不是前端原始 UI 状态。

示例 1，单选题：

```json
{
  "question_code": "Q1",
  "question_type": "single",
  "selected_option_code": "G11"
}
```

示例 2，多选题：

```json
{
  "question_code": "Q3",
  "question_type": "multi",
  "selected_option_codes": ["US", "UK", "HK"],
  "max_select": 3
}
```

示例 3，条件分支题：

```json
{
  "question_code": "Q8",
  "question_type": "branch_form",
  "branch_code": "A_LEVEL",
  "payload": {
    "school_gpa": "3.92",
    "rank_band": "TOP_10"
  }
}
```

#### `answer_display_text`

存用户气泡里展示的摘要文本。

示例：

- `G11（高二）`
- `美国、英国、中国香港`
- `计算机 / AI / 数据科学；数学 / 统计`

#### `version_no`

表示“这条答案当前生效时，对应的会话答案版本号”。

建议规则：

- 每成功提交一题
- `student_profile_guided_sessions.version_no + 1`
- 对应题目的答案行同步更新为最新 `version_no`

## 8. 新链路与正式业务表的关系

本期方案明确采用：

### 每题提交后的写库步骤

1. 更新 `student_profile_guided_sessions`
2. 更新 `student_profile_guided_answers`
3. 写入 `student_profile_guided_messages`
4. 基于当前累计答案，投影生成正式业务表 payload
5. 将 payload 立即写入正式业务表
6. 同步兼容镜像字段：
   - `student_basic_info.CTRY_CODE_VAL`
   - `student_basic_info.MAJ_CODE_VAL`
   - `student_basic_info.MAJ_INTEREST_TEXT`
7. 返回下一题

### 本期不采用

- 不额外引入中间快照表
- 不为了避免和旧链路冲突而延迟写正式业务表

即：

- 新链路每答一题，就真实更新正式业务表

## 9. 六维图生成策略

### 9.1 触发时机

以下情况都触发生成：

- 问卷全部答完
- 用户中途退出
- 聊天框关闭
- 页面关闭 / 标签页关闭
- 恢复会话时发现有新答案但尚未生成过最新结果

### 9.2 生成规则

判断条件：

- 如果 `student_profile_guided_sessions.version_no > last_generated_version_no`
- 则允许触发新一轮生成

生成成功后：

- 更新 `last_generated_version_no`
- 写 `student_profile_guided_results`
- 更新 `current_stage`

### 9.3 结果来源

新链路的六维图结果单独落在：

- `student_profile_guided_results`

不写旧的：

- `ai_chat_profile_results`

## 13. 新链路状态机建议

建议状态如下：

- `collecting`
- `build_ready`
- `scoring`
- `generated`
- `failed`

说明：

- `collecting`
  - 问卷收集中
- `build_ready`
  - 已具备生成六维图条件，等待或即将触发生成
- `scoring`
  - 正在生成六维图
- `generated`
  - 结果已生成
- `failed`
  - 本轮生成失败

## 15. 当前方案下的一个重要结果

由于本期采用“每题立即投影正式业务表”，所以：

- 用户答到一半时，正式业务表保存的是当前已答部分
- 中途退出时，可以直接基于当前正式业务表生成六维图
- 旧 AI 链路如果此时去读正式业务表，会看到当前这份半成品档案

这是本期已接受的产品口径。
